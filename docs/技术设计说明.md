# 智能知识管理平台 — 技术设计说明

> 本文档面向团队成员，说明"为什么这样做"背后的每一个决策。
> 不是 API 文档，是帮助新成员理解项目全貌的设计文档。

---

## 一、这个项目在解决什么问题

### 业务场景

企业知识散落在 PDF、Word、Excel、TXT 中，形成信息孤岛。传统的关键词搜索（如 Elasticsearch）无法理解语义——用户问"隐患点 A 降雨量到多少需要预警"，关键词搜索找不到"防汛阈值"这个表述。而且查一个实体，找不到它关联的规范、负责人、历史数据，需要人工多次查询。

### 我们的解法

构建一个 AI 知识库平台，用户上传文档后，可以：

1. **用自然语言提问**，AI 基于文档内容精准回答（RAG）
2. **看到回答的来源**，每个回答都标注引用了哪份文档的哪个章节
3. **发现关联实体**，查一个隐患点，自动关联到负责人、规范、监测数据（GraphRAG）
4. **开放 MCP 接口**，其他 AI Agent 可以通过标准协议查询知识库

## 二、整体架构

```
用户
  │
  ▼
┌──────────────────────────────────────────────┐
│  Next.js 前端（React + TypeScript + Tailwind） │
│  知识库管理 / AI 对话 / 图谱可视化              │
├──────────────────────────────────────────────┤
│  Next.js API Routes（服务端）                  │
│  /api/kb/*     知识库 CRUD                     │
│  /api/chat     RAG 对话                        │
├──────────────────────────────────────────────┤
│  业务逻辑层（lib/）                             │
│  chunker       文本切片                         │
│  embedding      向量化                          │
│  vector-store   向量检索（FileStore / ChromaDB）│
│  graph-store    知识图谱（Graphology）           │
│  entity-extractor 实体关系提取                   │
│  db             元数据持久化                     │
├──────────────────────────────────────────────┤
│  数据存储层                                     │
│  data/db.json       知识库 & 文档元数据          │
│  data/vectors.json  向量数据                     │
│  data/graph.json    知识图谱数据                 │
├──────────────────────────────────────────────┤
│  外部服务                                      │
│  智谱 GLM-4-Flash（LLM 问答）                   │
│  智谱 Embedding-3（文本向量化）                  │
└──────────────────────────────────────────────┘
```

### 为什么是单体架构而不是微服务

1. **这是演示项目**，不是生产系统。单体部署一个 `npm run dev` 就跑起来，面试官 clone 仓库就能看
2. **单进程部署**，可以直接上 Vercel，不需要 Docker/K8s
3. GraphRAG 方案调研过 LightRAG（Python），但集成需要跑 Python sidecar，部署复杂度翻倍。自建 TypeScript 版本保持架构简洁
4. 后续如果需要拆分，lib/ 层的模块边界清晰，拆分成本低

## 三、技术选型——每个决策的理由

### 3.1 框架：Next.js + React + TypeScript

| 选择 | 理由 |
|------|------|
| **Next.js** | App Router 支持 API Routes 和 SSR 一体化，前后端同一个项目，不需要单独的 Express 服务 |
| **React 19** | 生态最成熟，AI SDK（Vercel AI SDK）原生支持 React Hooks（useChat），流式对话开箱即用 |
| **TypeScript strict** | 类型安全避免运行时错误，接口设计先定义类型再写代码，团队协作时类型就是最好的文档 |

### 3.2 AI SDK：Vercel AI SDK v6

| 选择 | 理由 |
|------|------|
| **streamText** | 流式生成回答，用户不用等 LLM 全部生成完才看到结果，体验好 |
| **useChat** | 前端 Hook，自动管理消息列表、流式接收、停止/重新生成，省去手写 SSE 解析逻辑 |
| **generateObject** | 结构化输出（实体提取时用），配合 Zod schema 保证 LLM 返回的数据格式正确，不需要手写 JSON 解析 |
| **createOpenAI** | OpenAI 兼容接口适配器，智谱 GLM API 兼容 OpenAI 格式，只需改 baseURL 就能切换模型 |

#### 为什么现阶段用 AI SDK 而不是 LangChain.js

LangChain.js 是通用 LLM 编排框架，功能全但抽象层多。现阶段选择 AI SDK 的原因：

| 对比维度 | Vercel AI SDK | LangChain.js |
|---------|--------------|-------------|
| **学习曲线** | 低，API 直观 | 高，概念多（Chain/Agent/Memory/Tool/Retriever） |
| **流式对话** | 原生支持，useChat Hook 开箱即用 | 需要额外配置 streaming handler |
| **代码量** | 几十行搞定 RAG 对话 | 同样功能需要更多模板代码 |
| **前端集成** | React Hook 深度集成，消息/状态/停止自动管理 | 主要关注后端编排，前端需要自己对接 |
| **抽象程度** | 够用但不越权，你清楚每一行在干什么 | 高度抽象，出 bug 时排查链路长 |

**但 LangChain.js 在后续阶段有明确的价值**：

| 阶段 | 当前做法（手写） | LangChain.js 能优化什么 |
|------|----------------|----------------------|
| **多格式文档解析**（阶段二） | 自己用 pdfjs-dist/mammoth/exceljs 拼 | `PDFLoader` / `DocxLoader` / `CSVLoader` 开箱即用，已处理编码、分页、表格 |
| **检索质量优化**（阶段二） | 手写 RRF 融合 | `EnsembleRetriever` 多路检索 + `ContextualCompressionRetriever` 压缩冗余上下文 |
| **多轮对话记忆**（阶段二） | 拼接最近 2 轮历史，硬截断 500 字符 | `ConversationSummaryMemory` 自动压缩历史，保留关键信息不丢失 |
| **Agent 编排**（阶段三） | 无 | `Tool` + `AgentExecutor` 实现 ReAct 循环（思考→行动→观察→再思考），手写这个容易出错 |

**结论**：不是"不用 LangChain.js"，是"还不到时候"。阶段一 RAG 流程简单，AI SDK 更轻量透明；阶段二三按需引入 LangChain.js 的 DocumentLoader、Retriever、Agent 模块，按需引入而不是全量依赖。

### 3.3 LLM：智谱 GLM-5.1

#### 模型参数

| 属性 | 值 |
|------|---|
| 架构 | 754B MoE（混合专家），每 token 激活 40B 参数 |
| 上下文窗口 | 200K token（输入 + 输出） |
| 最大输出 | 128K token |
| Thinking 模式 | 可开关（开启后建议 temperature=1.0） |
| 定价 | 输入 $1.40/M token，输出 $4.40/M token |

#### 选型理由

| 选择 | 理由 |
|------|------|
| **GLM-5.1** | 智谱旗舰推理模型，中文能力强，200K 上下文足以处理长文档 |
| **OpenAI 兼容** | API 完全兼容 OpenAI 格式，可无缝切换到 DeepSeek/Qwen/OpenAI |
| **Function Calling** | 原生支持，generateObject 实体提取依赖此能力 |
| **Thinking 模式** | 复杂推理时可开启，提升回答质量；简单问答时关闭，降低延迟和成本 |

#### API 接入方式

**方式一：`zhipu-ai-provider`（推荐）**

智谱官方提供的 Vercel AI SDK 社区适配器，直接 `npm install zhipu-ai-provider`，无需手动配置 baseURL。

```typescript
import { createZhipu } from 'zhipu-ai-provider'

const zhipu = createZhipu({ apiKey: process.env.ZHIPU_API_KEY })

// 对话
const result = streamText({ model: zhipu('glm-5.1'), messages })

// 结构化输出（实体提取）
const { object } = generateObject({ model: zhipu('glm-5.1'), schema, prompt })

// Embedding（embedding-3，2048 维，可配置）
const embedModel = zhipu.embeddingModel('embedding-3', { dimensions: 256 })
```

**方式二：`@ai-sdk/openai` + 自定义 baseURL（现有代码）**

项目已在使用此方式，不需要改动即可切换模型名。

```typescript
const glm = createOpenAI({
  baseURL: 'https://open.bigmodel.cn/api/paas/v4',
  apiKey: process.env.ZHIPU_API_KEY,
})

streamText({ model: glm.chat('glm-5.1'), messages })  // 改模型名即可
```

#### 模型切换策略

不需要所有接口都用旗舰模型，按场景分配：

| 使用场景 | 推荐模型 | 理由 |
|---------|---------|------|
| **RAG 对话问答** | GLM-5.1（thinking 关闭） | 日常问答不需要推理，关闭 thinking 降低延迟和成本 |
| **实体关系提取** | GLM-5.1（thinking 关闭） | 结构化输出任务，thinking 模式下 schema-guided 不可用 |
| **复杂关联分析** | GLM-5.1（thinking 开启） | 需要推理能力的场景，如"分析隐患点 A 和 B 之间的风险差异" |
| **Embedding** | embedding-3 | 智谱唯一的 embedding 模型，2048 维可配置 |

#### 免费模型备选（开发测试用）

智谱提供多个免费模型，开发调试时可以切换以节省成本：

| 模型 | 价格 | 上下文 | 适用场景 |
|------|------|--------|---------|
| GLM-4.7-Flash | 免费 | 128K | 快速测试对话 |
| GLM-4-Flash | 免费 | 128K | 原有代码已验证 |
| GLM-4V-Flash | 免费 | 8K | 图片理解（后续扩展） |

#### 注意事项

1. **Thinking 模式与 generateObject 不兼容**：GLM-5.1 开启 thinking 时，schema-guided 结构化输出不可用。实体提取时必须关闭 thinking：`zhipu('glm-5.1').withConfig({ thinking: { type: 'disabled' } })`
2. **Embedding 仅限国内端点**：`open.bigmodel.cn` 支持 embedding API，国际端点 `api.z.ai` 不支持
3. **架构预留 Ollama 本地部署能力**：政企客户要求数据不出内网时，通过 Ollama 部署本地 Qwen2.5/DeepSeek 模型，只需切换 baseURL

### 3.4 向量存储：FileStore（JSON 文件）

| 选择 | 理由 |
|------|------|
| **FileStore（默认）** | 向量数据持久化到 `data/vectors.json`，零外部依赖，开发阶段最方便 |
| **ChromaDB（可选）** | 生产环境切换，通过环境变量 `VECTOR_STORE=chroma` 激活，接口不变 |

**为什么默认不用数据库**：演示项目的数据量（500 隐患点 + 20 规范文档）JSON 文件完全够用。向量数据本质上是高维浮点数组 + 余弦相似度计算，不需要 SQL 查询能力。

### 3.5 文档解析：pdfjs-dist + mammoth + exceljs

| 库 | 解决的问题 | 为什么选它 |
|----|-----------|-----------|
| **pdfjs-dist** | PDF 文本提取 | Mozilla 官方维护，中文 PDF 的 CJK 字体支持最好，npm 周下载量 165 万+ |
| **mammoth** | Word (.docx) 转 HTML/文本 | 轻量、专注语义正确（标题/段落/列表/表格），不做像素级排版——正好适合知识提取 |
| **exceljs** | Excel 读取 | SheetJS（xlsx）已停更 4 年，exceljs 积极维护，支持流式读取大文件、单元格样式、公式 |

**为什么不用 pdf-parse？** pdf-parse v2 做了 breaking change，而 pdfjs-dist 是 PDF 解析的行业标准（Chrome 浏览器内置的 PDF 渲染引擎就是它），中文支持更可靠。

### 3.6 知识图谱：Graphology（自建 TypeScript 版）

| 选择 | 理由 |
|------|------|
| **Graphology** | JavaScript 图数据库库，内置 BFS/DFS 遍历、图序列化/反序列化，TypeScript 支持好 |
| **JSON 持久化** | 与 FileStore 同一思路，`data/graph.json` 零依赖 |
| **自建而非 LightRAG** | LightRAG 是 Python 库，集成需要 Python sidecar → 部署复杂度翻倍 → 违背单体架构原则 |

**Graphology 解决了什么**：
- **图存储**：节点（实体）和边（关系）的 CRUD，支持有向图、多重图
- **图遍历**：BFS/DFS 从一个实体出发，发现所有关联实体（深度可控）
- **序列化**：`graph.export()` → JSON 存储，`Graph.from(data)` → 加载恢复

### 3.7 实体提取：AI SDK generateObject + Zod

**解决的问题**：从文本中自动提取实体（人名、地点、组织、文档名）和关系（负责、位于、引用）。

**为什么用 generateObject 而不是手写 Prompt + 正则解析**：
- LLM 返回的 JSON 格式不稳定，手写解析容易出错
- Zod schema 同时做两件事：① 约束 LLM 输出格式 ② 运行时类型校验
- `.describe()` 的文字会作为提取提示发给 LLM，不需要在 prompt 里写格式说明

**注意事项**：必须用 `@ai-sdk/openai`（非 `@ai-sdk/openai-compatible`），后者不支持 generateObject。如果默认模式失败，加 `mode: 'tool'` 强制使用 function calling。

### 3.8 图谱可视化：react-force-graph-2d

| 选择 | 理由 |
|------|------|
| **react-force-graph-2d** | HTML5 Canvas 渲染，性能好；API 极简——一个组件 + 一份数据即可渲染知识图谱 |
| **动态导入** | Next.js 中必须 `dynamic(() => import(...), { ssr: false })`，因为它依赖浏览器 Canvas API |

### 3.9 MCP 接口：@modelcontextprotocol/sdk

| 选择 | 理由 |
|------|------|
| **Stdio Transport** | 最简单的集成方式——Claude Desktop 直接调用知识库查询工具，不需要 HTTP 服务器 |
| **v1.29.0** | 官方 TypeScript SDK 已稳定，暴露 2-3 个工具（搜索知识库、查询关联实体、列出文档）即可 |

**MCP 解决什么问题**：知识库不是孤岛。通过 MCP 协议，任何 AI Agent 都能查询知识库——新人指引 Agent、出题 Agent、合规审查 Agent 等都可以通过标准接口接入。

## 四、数据存储设计

### 4.1 为什么不用传统数据库

这个项目的数据分两类：

| 数据类型 | 特点 | 存储方式 |
|---------|------|---------|
| **元数据**（知识库、文档列表） | 关系简单、查询固定 | `data/db.json` |
| **向量数据**（embedding） | 高维浮点数组，查询用余弦相似度 | `data/vectors.json` |
| **图谱数据**（实体、关系） | 图结构，查询用 BFS 遍历 | `data/graph.json` |

元数据没有复杂的关系查询（不需要 JOIN），向量数据不能用 SQL 查询（需要余弦相似度），图谱数据需要图遍历（不是表结构）。三者的查询模式都不适合关系型数据库，各自用最适合的存储方式。

**生产环境升级路径**：db.json → SQLite/PostgreSQL，vectors.json → ChromaDB/Milvus，graph.json → Neo4j。接口层已抽象，切换只需改实现类。

### 4.2 db.json — 元数据存储

```typescript
// data/db.json 结构
{
  "knowledgeBases": [
    { "id": "know_1", "name": "地灾规范", "description": "...", "docCount": 5, "createdAt": "..." }
  ],
  "documents": [
    { "id": "doc_1", "kbId": "know_1", "filename": "滑坡防治规范.txt", "chunkCount": 12, "uploadedAt": "...", "chunkIds": ["doc_1_0", "doc_1_1", ...] }
  ],
  "knowledgeBaseId": 1,   // 自增 ID 计数器
  "documentsId": 1         // 自增 ID 计数器
}
```

**设计要点**：
- `chunkIds` 记录每个文档产生的切片 ID，删除文档时能精确清理对应的向量数据
- 知识库和文档是一对多关系，通过 `kbId` 关联
- ID 格式 `know_{n}` / `doc_{n}` 而不是 UUID，方便调试和日志排查

### 4.3 vectors.json — 向量存储

```typescript
// data/vectors.json 结构
{
  "kb-know_1": {                    // collection 名 = kb-{知识库ID}
    "ids": ["doc_1_0", "doc_1_1"],  // 切片 ID
    "texts": ["切片文本...", "..."],  // 切片原文
    "vectors": [[0.12, -0.34, ...], [...]], // 1024 维向量
    "metas": [                       // 来源元数据（与 ids 一一对应）
      { "filename": "滑坡防治规范.txt", "heading": "第三章", "chunkIndex": 0 },
      null                           // 旧数据可能没有 metas，返回 null
    ]
  }
}
```

**设计要点**：
- 四个平行数组（ids/texts/vectors/metas）一一对应，索引相同的数据属于同一个切片
- 按 collection 分组（`kb-{知识库ID}`），不同知识库的向量数据隔离
- `metas` 字段可选，旧数据为 `null`（向后兼容）

### 4.4 graph.json — 知识图谱（规划中）

```typescript
// data/graph.json 结构（Graphology 导出格式）
{
  "options": { "type": "directed", "multi": false },
  "nodes": [
    { "key": "隐患点A", "attributes": { "name": "隐患点A", "type": "Location", "sourceChunks": ["doc_1_0"] } },
    { "key": "张三", "attributes": { "name": "张三", "type": "Person", "sourceChunks": ["doc_2_3"] } }
  ],
  "edges": [
    { "source": "隐患点A", "target": "张三", "attributes": { "relation": "RESPONSIBLE_FOR", "evidence": "..." } }
  ]
}
```

**设计要点**：
- 节点 `key` 用实体名称（而非 UUID），方便关键词匹配查询
- `sourceChunks` 回溯到向量切片，图查询和向量查询可以联动
- 边上的 `evidence` 记录支撑这条关系的原文片段，可解释

## 五、接口设计

### 5.1 设计原则

1. **RESTful 风格** — 资源名用名词，操作用 HTTP 方法（GET 查询、POST 创建、DELETE 删除）
2. **错误统一格式** — 所有错误返回 `{ error: string }`，前端统一处理
3. **流式接口用 SSE** — 对话接口返回 Server-Sent Events 流，不是一次性 JSON

### 5.2 接口清单

#### 知识库管理

| 方法 | 路径 | 功能 | 请求 | 响应 |
|------|------|------|------|------|
| GET | `/api/kb` | 获取知识库列表 | - | `KnowledgeBase[]` |
| POST | `/api/kb` | 创建知识库 | `{ name, description }` | `KnowledgeBase` |
| DELETE | `/api/kb/[id]` | 删除知识库（含向量数据） | - | `{ success: true }` |

#### 文档管理

| 方法 | 路径 | 功能 | 请求 | 响应 |
|------|------|------|------|------|
| GET | `/api/kb/[id]/docs` | 获取文档列表 | - | `Document[]` |
| POST | `/api/kb/[id]/upload` | 上传文档 | `FormData { file }` | `{ success: true }` |
| DELETE | `/api/kb/[id]/docs` | 删除文档（含向量） | `{ docId }` | `{ success: true }` |

#### AI 对话

| 方法 | 路径 | 功能 | 请求 | 响应 |
|------|------|------|------|------|
| POST | `/api/chat` | RAG 对话（流式） | `{ messages, kbId }` | SSE 流 |

### 5.3 核心接口详解

#### POST /api/chat — RAG 对话流程

这是系统最核心的接口，完整流程如下：

```
1. 接收请求 → 提取用户消息和历史对话
2. 构建搜索查询 → 拼接最近 2 轮对话 + 当前问题（多轮上下文优化）
3. 向量化查询 → 调用智谱 Embedding-3 API
4. 混合检索 → 向量搜索（语义匹配）+ 关键词搜索（精确匹配），RRF 融合排序
5. 过滤低相关性结果 → 余弦相似度 < 0.45 视为不相关（混合搜索跳过此步骤）
6. 构建上下文 → 编号引用格式 [1] [2] [3]...
7. 调用 LLM → GLM-4-Flash 流式生成回答
8. 附加来源元数据 → SourceRef[] 通过 messageMetadata 传给前端
9. 返回 SSE 流
```

**为什么 upload 用 XHR 不用 fetch？**

fetch API 不支持 `upload.onprogress` 事件，无法显示上传进度条。所以上传组件用 XMLHttpRequest。

## 六、前端设计

### 6.1 页面结构

整个应用只有一个页面，通过状态切换两个视图：

```

## 八、核心算法说明

### 8.1 文本切片（chunker.ts）

**为什么需要切片**：LLM 的上下文窗口有限（GLM-4-Flash 128K token），不能把整份文档塞进去。需要把文档切成小段，只把相关的段落注入上下文。

**切片策略**：
1. 先按 Markdown 标题（`# ~ ######`）切分
2. 小节（≤ 1000 字符）保持完整
3. 大节按 1000 字符切分，相邻切片重叠 100 字符（避免语义被切断）

**为什么是 1000 字符**：embedding 模型对 300-1500 字符的文本效果最好。太短丢失上下文，太长稀释语义。100 字符重叠确保边界处的语义连续。

### 8.2 混合检索（RRF 融合）

**为什么不能只用向量搜索**：向量搜索擅长语义匹配，但会遗漏精确关键词。例如用户搜"GLM-4-Flash"，向量搜索可能匹配到"大语言模型"但找不到模型名本身。

**混合检索流程**：
1. **向量搜索**：查询向量化 → 余弦相似度排序 → Top 5
2. **关键词搜索**：中文逐字分词 + 英文单词分词 → 匹配词频排序 → Top 5
3. **RRF 融合**：两路结果按公式合并 `score = Σ(1 / (60 + rank))`，取 Top 5

**为什么用 RRF 而不是简单加权**：向量分数和关键词分数的量级不同（余弦 0-1 vs 词频 0-1），直接加权不公平。RRF 只看排名不看分数，天然消除了量级差异。

### 8.3 多轮对话上下文

**问题**：用户追问"详细说说"，这个查询本身没有语义信息，向量化后检索不到任何相关内容。

**解法**：`buildSearchQuery` 拼接最近 2 轮对话 + 当前问题作为检索查询，截断到 500 字符避免稀释 embedding。

**为什么只影响检索不影响 LLM**：LLM 接收的是完整的原始对话消息（messages 数组），拼接只用于 embedding 输入。

src/
├── app/                        # Next.js App Router
│   ├── layout.tsx              # 根布局（字体、语言、meta）

│   ├── globals.css             # Tailwind CSS 入口
│   └── api/                    # API 路由
│       ├── chat/route.ts       # RAG 对话（核心接口）
│       └── kb/
│           ├── route.ts        # 知识库 CRUD
│           └── [id]/
│               ├── route.ts    # 删除知识库
│               ├── docs/route.ts    # 文档列表 + 删除
│               └── upload/route.ts  # 文档上传
├── components/                 # React 组件
│   ├── ChatPanel.tsx           # 对话面板（useChat + 来源标签）
│   └── Toast.tsx               # 通知提示
└── lib/                        # 业务逻辑（与框架无关）
    ├── types.ts                # 共享类型定义
    ├── chunker.ts              # 文本切片
    ├── embedding.ts            # 向量化（智谱 API）
    ├── db.ts                   # 元数据持久化（db.json）
    ├── logger.ts               # 日志
    └── vector-store/           # 向量存储
        ├── types.ts            # VectorStore 接口
        ├── index.ts            # 工厂函数（环境变量切换实现）
        ├── file-store.ts       # JSON 文件实现 + 混合检索
        └── chroma-store.ts     # ChromaDB 实现

data/                           # 运行时数据（gitignore）
├── db.json                     # 知识库 & 文档元数据
├── vectors.json                # 向量数据
└── app.log                     # 应用日志
```

**设计原则**：
- `lib/` 与框架无关，可以单独测试
- `components/` 只管 UI，不直接调 API
- `app/api/` 只做参数校验 + 调用 `lib/`，不写业务逻辑
- `data/` 全部 gitignore，clone 后首次运行自动生成
