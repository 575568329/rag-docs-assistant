# 智能知识管理平台 — 前端设计方案

> 基于 ChatGPT / Perplexity / Dify / Neo4j Bloom / Obsidian 等产品的 UI 模式调研，
> 结合项目实际需求设计。

---

## 一、页面结构

### 1.1 为什么是三个页面

参考 Dify、FastGPT、RAGFlow 等 AI 知识库产品的布局，它们都采用"左侧导航 + 右侧内容区"的仪表盘模式。我们的项目有三个功能重心，拆成三个页面而非挤在一起：

| 页面 | 核心功能 | 参考产品 |
|------|---------|---------|
| **对话页** | AI 问答、来源追溯、关联实体跳转 | ChatGPT（对话区）+ Perplexity（来源引用） |
| **数据管理页** | 知识库 CRUD、文档上传/删除/列表 | Dify（知识库管理面板） |
| **知识图谱页** | 图谱可视化、实体浏览、关系探索 | Obsidian Graph View + Neo4j Bloom |

**为什么不做成一个页面**：对话、数据管理、图谱探索的交互模式完全不同（输入框 vs 文件上传 vs 拖拽缩放），硬塞一个页面会让每个功能都施展不开。

### 1.2 整体布局

```
┌──────────────────────────────────────────────────────┐
│  顶部导航栏（固定）                                    │
│  [💬 对话]  [📁 数据管理]  [🔗 知识图谱]              │
├──────────┬───────────────────────────────────────────┤
│          │                                           │
│  左侧边栏  │            主内容区                       │
│  知识库选择 │                                           │
│          │                                           │
│  ○ 全部   │                                           │
│  ● 地灾规范│                                           │
│  ○ 教育知识│                                           │
│          │                                           │
│          │                                           │
├──────────┴───────────────────────────────────────────┤
│  （无底栏，内容区撑满）                                 │
└──────────────────────────────────────────────────────┘
```

**参考**：ChatGPT 的左侧对话列表 + 右侧对话区；Dify 的左侧应用列表 + 右侧工作区。

**为什么用顶部导航而不是侧边 tab**：三个页面功能差异大，顶部导航更清晰地表达"你在哪里"。侧边栏留给知识库选择器——这是跨页面共享的状态。

## 二、对话页（默认首页）

### 2.1 布局

```
┌──────────┬───────────────────────────────────────────┐
│  知识库   │                                           │
│  选择器   │         对话消息区                         │
│          │                                           │
│  ○ 全部   │  用户: 隐患点A的预警阈值是多少？            │
│  ● 地灾   │                                           │
│  ○ 教育   │  AI: 根据《地质灾害防治条例》第三章，        │
│          │      隐患点A所在区域为橙色预警区，            │
│          │      24小时降雨量达到50mm需启动三级响应。¹     │
│          │                                           │
│          │  来源: [1] 防治条例-第三章 0.89              │
│          │       [2] XX市应急预案 0.76                  │
│          │                                           │
│          │  关联实体: [隐患点A] [橙色预警] [XX市] [张三]  │
│          │        ↗ 点击可跳转知识图谱                  │
│          │                                           │
│  ──────  ├───────────────────────────────────────────┤
│  知识库   │  [在此输入问题...]                [发送]     │
│  详情     │  当前知识库: 地灾规范 | 切换                │
└──────────┴───────────────────────────────────────────┘
```

### 2.2 核心交互

#### 知识库选择

| 选项 | 行为 | 为什么 |
|------|------|--------|
| **全部** | 搜索所有知识库的向量数据，合并结果 | 用户不确定信息在哪个知识库时使用 |
| **指定知识库** | 只搜索该知识库的向量数据 | 精确检索，避免其他知识库的干扰 |

**参考**：Perplexity 的 Focus 模式（选择搜索源）、Copilot Studio 的数据源配置。

**实现方式**：左侧边栏的选中状态通过 URL query 参数传递（`?kb=all` 或 `?kb=know_1`），刷新页面后保持状态。

#### 来源引用

参考 **Perplexity** 的引用模式（业界公认最好的来源展示方式）：

```
AI 回复中的引用标记（上标数字）：
  "降雨量达到50mm需启动三级响应。¹²"

来源卡片（回复下方展开）：
  ┌─────────────────────────────────────┐
  │  📄 来源                             │
  │  [1] 防治条例-第三章  0.89  [查看原文]│
  │  [2] XX市应急预案      0.76  [查看原文]│
  └─────────────────────────────────────┘
```

**为什么用上标数字而不是 [1] 方括号**：Perplexity、Google Gemini、Bing Chat 都用上标数字，视觉上更干净，不打断阅读流。我们保留方括号（已实现），后续可以优化。

#### 关联实体跳转

AI 回复下方的关联实体标签组，点击可跳转到知识图谱页面：

```
关联实体: [隐患点A] [橙色预警] [张三] [防治条例]
           ↓ 点击
           跳转到知识图谱页，相机动画聚焦到该节点
```

**实现**：点击标签 → 路由跳转 `/graph?focus=隐患点A`，图谱页面读取 focus 参数，动画聚焦到对应节点。

### 3.2 核心交互

| 交互 | 实现 | 参考 |
|------|------|------|
| **知识库切换** | 左侧选择器切换，右侧内容区刷新 | Dify 知识库管理 |

## 四、知识图谱页

### 4.1 布局

参考 **Obsidian Graph View** + **Neo4j Bloom** 的设计：

```
┌──────────┬───────────────────────────────────────────┐
│  知识库   │  [搜索实体...]            [布局] [全屏]    │
│  选择器   │  ┌─────────────────────────────────────┐  │
│          │  │                                     │  │
│          │  │          图谱画布                     │  │
│          │  │     (react-force-graph-2d)           │  │
│          │  │                                     │  │
│          │  │        ● ── 隐患点A ── ● 张三        │  │
│          │  │       / \              \             │  │
│          │  │      ●   ●              ● 防治规范    │  │
│          │  │     滑坡  橙色预警                    │  │
│          │  │                                     │  │
│          │  │                          [小地图]    │  │
│          │  └─────────────────────────────────────┘  │
│          │                                           │
│          │  ┌─ 实体详情面板（点击节点后展开）──────────┐│
│          │  │  隐患点A  |  Location                    ││
│          │  │  属性: 区域=XX市, 级别=省级              ││
│          │  │  关联: 4 个实体, 3 份文档                 ││
│          │  │  来源文档: [防治条例] [应急预案]          ││
│          │  └────────────────────────────────────────┘│
└──────────┴───────────────────────────────────────────┘
```

### 4.2 核心交互

| 交互 | 行为 | 参考 |
|------|------|------|
| **点击节点** | 高亮该节点及其邻居，展开详情面板 | Obsidian（hover 高亮连线）+ Neo4j Bloom（Inspector 面板） |
| **双击节点** | 扩展一跳邻居（BFS 深度 +1） | Neo4j Bloom 右键 Expand |
| **悬停节点** | 高亮该节点的所有连线，其余变暗 | Obsidian Graph View |
| **拖拽节点** | 手动调整节点位置 | 所有图谱工具 |
| **滚轮缩放** | 缩放画布 | 所有图谱工具 |
| **搜索** | 输入实体名，匹配节点高亮，相机动画聚焦 | Neo4j Bloom 搜索栏 |
| **URL 聚焦** | 从对话页跳转 `?focus=隐患点A`，自动聚焦 | 本项目自定义 |
| **实体详情面板** | 点击节点后底部/侧边展开，显示属性、关联、来源文档 | Neo4j Bloom Inspector |
| **颜色分组** | 按实体类型着色（Person 蓝色、Location 绿色、Document 橙色） | Obsidian Groups |

### 4.3 渐进式加载策略

**关键原则：永远不要一次加载全部图谱。**

参考 Obsidian 的 local graph vs global graph 模式：

| 场景 | 加载策略 | 节点数 |
|------|---------|--------|
| **默认进入** | 展示实体类型统计 + 搜索入口 | 0（空白画布 + 引导） |
| **搜索实体** | 加载目标实体 + 1 跳邻居 | ~10-20 |
| **双击扩展** | 在当前基础上再扩展 1 跳 | +10-30 |
| **从对话页跳转** | 加载目标实体 + 2 跳邻居 | ~20-50 |

**为什么不全量加载**：500 个节点同时渲染会形成"毛球"（hairball），用户看不清任何关系。渐进式加载让用户聚焦于感兴趣的局部子图。

## 五、样式设计

### 5.1 设计风格

参考 **Linear**（项目管理工具）和 **Perplexity** 的设计语言：极简、克制、留白充分。

| 维度 | 选择 | 理由 |
|------|------|------|
| **整体风格** | 极简白底，低信息密度 | 知识库工具不是社交产品，不需要花哨的视觉，内容本身是主角 |
| **圆角** | 12px（组件）/ 8px（按钮/输入框） | 参考 Linear 的圆角规范，现代但不夸张 |
| **阴影** | 极少使用，用边框代替 | 阴影在白底上容易显脏，1px 浅灰边框更干净 |
| **图标** | 线性图标（Lucide React） | 与 Tailwind 生态一致，轻量不抢视觉 |
| **字体** | 系统字体栈（Geist Sans 优先） | 不需要额外加载中文字体，性能好 |

### 5.2 配色方案

参考 Perplexity 和 Linear 的配色——中性灰为主，蓝色作为唯一强调色：

| 用途 | 色值 | 说明 |
|------|------|------|
| **页面背景** | `#FFFFFF` | 纯白 |
| **侧边栏背景** | `#F9FAFB` (gray-50) | 微灰，与主内容区区分 |
| **主文字** | `#111827` (gray-900) | 深灰近黑 |
| **次要文字** | `#6B7280` (gray-500) | 描述、时间戳 |
| **强调色（主色）** | `#2563EB` (blue-600) | 按钮、选中态、链接、来源标签 |
| **强调色（hover）** | `#1D4ED8` (blue-700) | 交互反馈 |
| **成功** | `#059669` (emerald-600) | 上传完成、操作成功 |
| **危险** | `#DC2626` (red-600) | 删除按钮、错误提示 |
| **边框** | `#E5E7EB` (gray-200) | 卡片边框、分割线 |
| **输入框焦点** | `#2563EB` 2px 描边 | blue-600，与主色一致 |

### 5.3 图谱配色（按实体类型）

| 实体类型 | 颜色 | 色值 |
|---------|------|------|
| Person（人物） | 蓝色 | `#3B82F6` |
| Location（地点） | 绿色 | `#10B981` |
| Organization（组织） | 紫色 | `#8B5CF6` |
| Document（文档） | 橙色 | `#F59E0B` |
| DisasterType（灾害类型） | 红色 | `#EF4444` |
| Regulation（法规） | 靛蓝 | `#6366F1` |
| Equipment（设备） | 青色 | `#06B6D4` |
| Other（其他） | 灰色 | `#6B7280` |

### 5.4 响应式断点

| 断点 | 宽度 | 侧边栏行为 |
|------|------|-----------|
| **桌面** | ≥ 1024px | 侧边栏固定显示（240px 宽） |
| **平板** | 768-1023px | 侧边栏可收起（汉堡菜单触发） |
| **手机** | < 768px | 侧边栏隐藏，底部导航栏替代顶部导航 |

**但**：演示项目以桌面端为主，手机端只保证不报错即可，不需要精细适配。

## 六、技术实现方案

### 6.1 路由设计

```
app/
├── layout.tsx              # 根布局（侧边栏 + 顶部导航）
├── page.tsx                # / → 重定向到 /chat
├── chat/
│   └── page.tsx            # 对话页
├── data/
│   └── page.tsx            # 数据管理页
├── graph/
│   └── page.tsx            # 知识图谱页
└── api/                    # API 路由（不变）
```

### 6.2 共享状态

知识库选择器在三个页面之间共享，用 URL 参数而非全局状态管理：

```
/chat?kb=know_1     → 对话页，选中"地灾规范"知识库
/data?kb=know_1     → 数据管理页，展示该知识库的文档
/graph?kb=know_1&focus=隐患点A → 图谱页，聚焦到隐患点A
```

**为什么用 URL 参数不用 Zustand/Context**：URL 可分享、可刷新、可回退，天然支持浏览器导航行为。

### 6.3 对话页到图谱页的跳转

```typescript
// ChatPanel 中点击关联实体标签
const handleEntityClick = (entityName: string) => {
  router.push(`/graph?kb=${currentKbId}&focus=${encodeURIComponent(entityName)}`)
}

// GraphPage 中读取 focus 参数，动画聚焦
const searchParams = useSearchParams()
const focusEntity = searchParams.get('focus')

useEffect(() => {
  if (focusEntity && fgRef.current) {
    const node = graph.findNode(focusEntity)
    if (node) {
      fgRef.current.cameraPosition(
        { x: node.x, y: node.y },  // 相机位置
        node,                        // 看向的目标
        1000                         // 动画时长 1 秒
      )
    }
  }
}, [focusEntity])
```

## 七、设计方案验证

### 7.1 与大厂方案的对比

| 维度 | ChatGPT | Perplexity | Dify | 本方案 |
|------|---------|-----------|------|--------|
| **导航方式** | 左侧对话列表 | 顶部搜索栏 | 左侧应用列表 | 顶部导航 + 左侧知识库选择器 |
| **知识库选择** | 无（单知识库） | 无（全网搜索） | 配置时绑定 | 对话时动态切换（含"全部"选项） |
| **来源展示** | 底部引用链接 | 上标数字 + 卡片 | 不展示 | 上标数字 + 来源卡片 + 分数 |
| **图谱可视化** | 无 | 无 | 无 | 独立页面 + 实体跳转 |
| **页面数量** | 1 | 1 | 多页面仪表盘 | 3 个页面 |

### 7.2 是否合理

| 检查项 | 结论 | 说明 |
|--------|------|------|
| **三个页面是否过多** | 合理 | Dify/FastGPT 也是多页面，每个功能有独立空间 |
| **对话页做首页是否合理** | 合理 | ChatGPT/Perplexity 首页都是对话，符合用户预期 |
| **知识库选择器全局共享** | 合理 | 比每个页面单独选择更高效 |
| **对话 → 图谱跳转** | 合理 | Neo4j LLM Graph Builder 已验证此交互模式 |
| **渐进式图谱加载** | 合理 | Obsidian local graph 的成功实践，避免"毛球"问题 |
| **配色方案** | 合理 | Perplexity/Linear 风格极简白底，适合知识工具 |

### 7.3 潜在问题

| 问题 | 影响 | 应对 |
|------|------|------|
| **图谱页冷启动时空白** | 用户不知道干什么 | 展示引导文字："搜索一个实体开始探索" + 实体类型统计 |
| **全部知识库搜索可能慢** | 搜索多个 collection 增加延迟 | 并行查询 + 前端 loading 状态 |
| **手机端图谱不可用** | 触摸操作不友好 | 图谱页提示"请在桌面端使用"，不强制适配 |
