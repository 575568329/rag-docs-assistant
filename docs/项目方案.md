## 一、项目背景与问题定义

### 1.1 真实业务场景

**场景一：地质灾害防治（省级）**

| 数据维度 | 规模 | 说明 |
|---------|------|------|
| 隐患点 | ~10 万个 | 分布在全省各区域 |
| 关联人员 | ~5-6 万人 | 责任人、监测员、管理人员 |
| 规范文档 | 数百到上千份 | 国标、省标、行业规范、年度政策 |
| 监测数据 | 持续增长 | 周报、月报、汛期数据，按隐患点关联 |
| 文件格式 | 混合 | Excel（导出数据）、PDF（规范）、Word（报告） |

**场景二：教育培训**

| 数据维度 | 说明 |
|---------|------|
| 知识点体系 | 学段 → 学科 → 知识点 → 题目，层级结构明确 |
| 数据来源 | 结构化数据（题库）+ 非结构化文档（教材、教案） |
| 核心需求 | 知识关联发现、快速检索、自动出题 |

### 1.2 行业共性痛点

以下痛点跨行业存在（政府、教育、医疗、法律、金融）：

| 痛点 | 具体表现 | 影响规模 |
|------|---------|---------|
| **信息孤岛** | 知识散落在 PDF/Word/Excel/邮件中，无法统一检索 | 全行业 |
| **语义鸿沟** | Elasticsearch 等关键词搜索无法理解"这个隐患点降雨量到多少需要预警"这类语义问题 | 全行业 |
| **关联缺失** | 查一个隐患点，找不到关联的规范、历史数据、负责人，需要人工多次查询 | 政府/企业 |
| **重复答疑** | 新人反复问同样问题，高级工程师时间被占用 | 全行业 |
| **知识流失** | 员工离职带走隐性知识，文档沉淀不下来 | 全行业 |
| **数据安全** | 政府/金融数据不能放到第三方 SaaS，必须私有化部署 | 政府/金融 |

### 1.3 为什么现有方案不够用

| 现有方案 | 局限性 |
|---------|--------|
| **Elasticsearch** | 关键词匹配，无法理解语义。搜"降雨量预警"可能找不到"防汛阈值" |
| **Confluence / 飞书** | 第三方 SaaS，数据不在自己手里；搜索仍是关键词模式 |
| **传统知识库** | 文档管理可以，但没有 AI 问答能力，用户体验差 |
| **通用大模型** | 不了解企业内部知识，会产生幻觉，无法引用来源 |

## 二、大厂实现方案调研

### 2.1 Microsoft 365 Copilot — 企业知识图谱 + RAG

Microsoft 365 Copilot 是目前全球规模最大的企业级 AI 知识系统之一，其核心架构直接验证了 "GraphRAG + VectorRAG" 双引擎方案的可行性。

**架构要点**：

| 层级 | 技术方案 | 与本方案的对应 |
|------|---------|--------------|
| **语义检索** | Microsoft Search + Vector Embedding | 我们的 VectorRAG（智谱 Embedding + FileStore） |
| **知识图谱** | Microsoft Graph（组织结构、文档关系、沟通网络） | 我们的 Graphology 知识图谱（JSON 持久化） |
| **数据接入** | SharePoint / Teams / Outlook / OneDrive 统一连接器 | 我们的多格式文档解析管道 |
| **安全控制** | Microsoft Purview（数据防泄漏、权限继承） | 我们的 RBAC 权限管理 |
| **LLM 层** | GPT-4 + 企业数据注入（Grounding） | 我们的智谱 GLM API + Prompt 注入上下文 |

**关键启示**：
1. **GraphRAG 不是概念，是生产实践** — Microsoft 在 2024 年开源了 [GraphRAG 库](https://github.com/microsoft/graphrag)，用 LLM 从文本中自动提取实体关系构建知识图谱，已在医疗、金融、法律领域部署
2. **数据源统一接入是核心挑战** — Copilot 支持超过 20 种数据源，每种都有独立的解析器。我们的多格式管道（PDF/Word/Excel/TXT/邮件）是缩小版但思路一致
3. **权限继承至关重要** — 企业环境中，用户只能看到自己有权限的文档，搜索结果需要过滤。这是我们不能忽略的

**参考资料**：
- [Microsoft GraphRAG GitHub](https://github.com/microsoft/graphrag)
- [Microsoft 365 Copilot Architecture — Microsoft Docs](https://learn.microsoft.com/en-us/copilot/microsoft-365/microsoft-365-copilot-overview)
- [Microsoft GraphRAG Paper — arXiv 2404.16135](https://arxiv.org/abs/2404.16135)

### 2.2 Google Vertex AI RAG Engine — "双脑" 托管架构

Google 在 2025 年推出了 Vertex AI RAG Engine，提供完全托管的 RAG 服务，其架构设计对中小团队有很强的参考价值。

**架构特点**：

```
Google Vertex AI RAG Engine
├── RAG Engine（语义检索引擎）
│   ├── 文档解析（PDF/HTML/DOCX 自动处理）
│   ├── 分块策略（递归分块 / 句子级分块）
│   └── 向量存储 + 检索
├── AlloyDB（结构化数据引擎）
│   ├── 保存原始文档元数据
│   ├── 保存实体关系数据
│   └── 支持混合查询（SQL + 向量）
└── Vertex AI Search（统一查询入口）
    ├── 语义搜索
    ├── 关键词搜索
    └── 混合排序（类似我们的 RRF）
```

**与本方案的对比**：

| 维度 | Google 方案 | 本方案 | 差距评估 |
|------|-----------|--------|---------|
| 向量检索 | Vertex AI Vector Search | FileStore / ChromaDB | 功能等价，规模差距大（Google 十亿级） |
| 知识图谱 | AlloyDB + Knowledge Graph | Graphology（JSON 持久化） | 演示级足够，生产环境可切换 Neo4j |
| 分块策略 | 自动分块（多策略） | 递归字符分块 | 需要优化：增加语义分块 |
| 混合检索 | 内置混合排序 | RRF 融合（已实现） | 思路一致 |
| 部署方式 | 全托管云服务 | 本地私有化部署 | **我们的优势：数据不出内网** |

**关键启示**：
1. Google 的 "双脑" 架构（向量引擎 + 关系引擎）验证了我们的双引擎设计方向
2. AlloyDB 同时承担向量存储和结构化查询，我们的 Graphology + FileStore 是更轻量的等价方案
3. Google 的分块策略更智能（语义分块），我们后续需要升级

### 2.3 开源 GraphRAG 项目对比

| 项目 | 核心特点 | 适用场景 | 与本方案关系 |
|------|---------|---------|------------|
| **[LightRAG](https://github.com/HKUDS/LightRAG)** | 双层检索（实体级 + 关系级），支持增量更新，Ollama 兼容 | 中小规模知识图谱构建 | **参考其核心思路，自建 TS 版本** |
| **[Microsoft GraphRAG](https://github.com/microsoft/graphrag)** | 社区检测 + 全局摘要，适合宏观分析 | 大规模文档集的全局性问题 | 太重，不适合我们的场景 |
| **[FalkorDB GraphRAG-SDK](https://github.com/FalkorDB/GraphRAG-SDK)** | 生产级 GraphRAG SDK，Redis-based | 需要低延迟的生产环境 | 参考其生产化经验 |
| **[ApeRAG](https://github.com/ApeRAG/aperag)** | 多模态 + MCP 支持，文档理解能力强 | 多格式复杂文档处理 | 参考其 MCP 集成方式 |
| **[RAGFlow](https://github.com/infiniflow/ragflow)** | 深度文档理解，模板化分块 | 高精度文档检索 | 竞品参考，我们走 GraphRAG 差异化 |

**选型结论：参考 LightRAG 思路，纯 TypeScript 自建**

经调研验证，LightRAG 是 Python 库，无 JS/TS SDK。集成需要 Python sidecar，会破坏单进程部署。
因此采用路线：参考 LightRAG 核心思路（实体提取 + 图谱构建 + 图遍历检索），用 Graphology + AI SDK generateObject 在 TypeScript 中自建轻量版。

| 评估维度 | Microsoft GraphRAG | LightRAG（Python） | 自建 TS 版 | 最终选择 |
|---------|-------------------|-------------------|-----------|---------|
| 语言 | Python | Python | **TypeScript** | 自建 |
| 部署复杂度 | 高（多依赖） | 中（需 sidecar） | **低（单进程）** | 自建 |
| 核心能力 | 社区检测 + 全局摘要 | 双层检索 + 增量更新 | 实体提取 + BFS 图遍历 | 演示够用 |
| 适用规模 | 10 万+ 文档 | 1 万以内 | **1 万以内** | 匹配 |

## 三、技术方案

### 3.1 系统架构

```
┌─────────────────────────────────────────────────────┐
│                    用户层（三页面）                    │
│  对话页（首页）/ 数据管理页 / 知识图谱页               │
│  左侧知识库选择器（全局共享）+ 顶部导航                │
├─────────────────────────────────────────────────────┤
│                  MCP 开放接口层                       │
│   MCP Server — 暴露查询能力，供其他 Agent 调用         │
├─────────────────────────────────────────────────────┤
│                  AI 检索引擎层                        │
│   GraphRAG（知识图谱关联发现）+ VectorRAG（语义精准检索）│
├─────────────────────────────────────────────────────┤
│                  数据处理层                           │
│   文档解析（PDF/Word/Excel/TXT）→ LLM 实体关系提取      │
├─────────────────────────────────────────────────────┤
│                  数据存储层                           │
│   知识图谱（Graphology JSON）+ 向量库 + 文档元数据库     │
├─────────────────────────────────────────────────────┤
│                  模型层                              │
│   GLM-5.1（754B MoE）+ embedding-3（智谱）            │
└─────────────────────────────────────────────────────┘
```

### 3.2 核心技术选型

| 模块 | 技术方案 | 选型理由 |
|------|---------|---------|
| **知识图谱** | JSON 文件存储（邻接表）+ LLM 自动提取实体关系 | 演示级数据量无需 Neo4j，零外部依赖；生产环境可无缝切换 |
| **语义检索** | 智谱 Embedding API + FileStore（JSON 持久化） | 已实现并验证，轻量够用 |
| **GraphRAG** | 纯 TypeScript 自建 | 无 Python 依赖，单进程部署，参考 LightRAG 核心思路简化实现 |
| **文档解析** | pdfjs-dist + mammoth + exceljs | PDF（pdfjs-dist，最佳中文支持）、Word（mammoth）、Excel（exceljs，SheetJS 已停更） |
| **LLM** | 智谱 GLM-5.1（754B MoE，200K 上下文） | 旗舰推理模型，中文能力最强；Thinking 模式可开关；API 兼容 OpenAI 格式 |
| **Embedding** | 智谱 embedding-3（2048 维，可配置） | 支持维度配置，中文效果好 |
| **前端** | Next.js App Router + React + TypeScript + Tailwind CSS | 三页面架构（对话/数据/图谱），zhipu-ai-provider 接入智谱 API |
| **图谱可视化** | react-force-graph-2d + Graphology | Canvas 渲染，渐进式加载，搜索聚焦，实体类型着色 |
| **MCP 接口** | @modelcontextprotocol/sdk（v1.x Stdio Transport） | 官方 TS SDK 已稳定，本地 Claude Desktop 直接调用，零部署成本 |

### 3.3 双引擎检索设计

传统 RAG 只解决"找到相关文本"的问题，不解决"找到关联实体"的问题。我们采用双引擎：

```
用户提问: "隐患点 A 的预警阈值是多少？"
                │
        ┌───────┴───────┐
        │               │
   VectorRAG        GraphRAG
   语义精准匹配      关联实体发现
        │               │
  找到包含"预警      找到隐患点A节点
  阈值"的文本段落     → 关联到灾害类型
                      → 关联到区域规范
                      → 关联到历史数据
                      → 关联到负责人
        │               │
        └───────┬───────┘
                │
          结果融合 + LLM 生成
                │
          回答 + 来源引用 + 关联图谱
```

| 检索场景 | 用哪个引擎 | 示例 |
|---------|-----------|------|
| 精确问答 | VectorRAG | "降雨量到多少需要预警？" |
| 关联发现 | GraphRAG | "隐患点A涉及哪些规范和人员？" |
| 综合查询 | 双引擎融合 | "隐患点A的历史汛期数据和对应预警标准" |

### 3.4 数据标准化管道

用户上传原始文件，系统自动完成全流程，无需用户改格式：

```
Excel（隐患点/人员数据）
  → exceljs 解析为结构化数据
  → 直接写入知识图谱（实体+关系）
  → 同时写入文档元数据库

PDF/Word（规范/报告）
  → 文本提取（pdfjs-dist / mammoth）
  → LLM 提取实体和关系
  → 写入知识图谱 + 切片向量化

TXT/MD（备注/笔记）
  → 直接切片
  → 向量化存入向量库

邮件数据（Exchange / IMAP）
  → 邮件头解析（发件人/收件人/时间/主题）
  → 正文提取（HTML 邮件转 Markdown）
  → 附件自动进入对应解析管道（PDF/Word/Excel）
  → 邮件通信关系写入知识图谱（人物 → 沟通网络）
  → 正文切片向量化

代码仓库（Git）
  → AST 解析提取函数签名、类定义、接口定义
  → 代码注释和文档字符串提取
  → 依赖关系分析（import/call graph）写入知识图谱
  → 代码片段切片向量化
```

#### 邮件数据接入详细方案

**为什么需要接入邮件？**

企业中大量隐性知识存在于邮件中：项目决策记录、技术方案讨论、客户需求确认、会议纪要转发等。这些邮件是知识图谱中"人物关系"和"决策上下文"的重要数据源。

**接入方式**：

| 方式 | 适用场景 | 实现难度 | 说明 |
|------|---------|---------|------|
| **Microsoft Graph API** | 企业使用 Exchange / Outlook | 中 | OAuth 2.0 认证，支持邮件/日历/联系人，企业推荐 |
| **IMAP 协议** | 通用邮箱（Gmail/QQ 等） | 低 | 标准协议，`node-imap` 库即可实现 |
| **EML/PST 文件导入** | 历史邮件批量导入 | 低 | 用户导出邮件文件后上传，系统解析 |

**数据提取流程**：

```
邮件接入
├── 1. 连接（Graph API / IMAP / 文件导入）
├── 2. 解析
│   ├── 邮件头 → 元数据（发件人、收件人、时间、主题、线程ID）
│   ├── 正文 → HTML 转 Markdown（cheerio + turndown）
│   └── 附件 → 按文件类型分发到对应解析管道
├── 3. 实体提取
│   ├── 人物实体（发件人/收件人 → 组织节点）
│   ├── 时间线（邮件线程 → 决策时间线）
│   └── 关键词/项目名 → LLM 自动提取
├── 4. 知识图谱写入
│   ├── Person -[EMAILED]-> Person（通信关系）
│   ├── Person -[MENTIONED]-> Project（项目关联）
│   └── Email -[CONTAINS]-> Decision/Requirement（决策记录）
└── 5. 向量化
    └── 正文切片 → 向量库（支持语义检索邮件内容）
```

**隐私与合规**：
- 邮件过滤：仅接入指定文件夹/标签的邮件（如"项目文档"、"技术方案"）
- 敏感信息脱敏：自动识别并遮蔽身份证号、银行卡号、密码等
- 权限继承：邮件的访问权限与原邮箱权限一致

#### 代码知识沉淀方案

**为什么代码也需要沉淀到知识库？**

团队代码是隐性知识的最大载体：架构决策、设计模式、性能优化技巧、踩坑经验都藏在代码中。新人需要数月才能熟悉代码库，通过代码知识库可以大幅缩短这个周期。

**代码知识提取流程**：

```
代码仓库接入
├── 1. 仓库连接（Git Clone / GitHub API / Gitea Webhook）
├── 2. 代码解析
│   ├── AST 分析（TypeScript → ts-morph / Python → ast）
│   │   ├── 函数签名 + 参数类型 + 返回值
│   │   ├── 类定义 + 属性 + 方法
│   │   ├── 接口/类型定义
│   │   └── 导入/导出关系
│   ├── 文档提取
│   │   ├── JSDoc / TSDoc 注释
│   │   ├── README.md
│   │   └── CHANGELOG.md
│   └── 依赖分析
│       ├── import 依赖图（模块间调用关系）
│       └── 函数调用图（caller → callee）
├── 3. 知识图谱写入
│   ├── Module -[IMPORTS]-> Module（模块依赖）
│   ├── Function -[CALLS]-> Function（调用关系）
│   ├── Class -[EXTENDS]-> Class（继承关系）
│   ├── Function -[BELONGS_TO]-> Module（归属关系）
│   └── Developer -[AUTHORED]-> Commit（贡献者关联）
├── 4. 向量化
│   ├── 函数/类级别的代码片段 → 向量库
│   └── 注释和文档 → 向量库
└── 5. 智能问答场景
    ├── "用户认证的流程是什么？" → 返回代码调用链 + 注释说明
    ├── "谁负责支付模块？" → 知识图谱查询贡献者
    └── "这个 API 的参数说明？" → 函数签名 + JSDoc
```

### 3.5 MCP 开放接口

MCP（Model Context Protocol）是 Anthropic 制定的开放标准，2026 年已成为 AI Agent 通信的行业规范。

```
知识库平台作为 MCP Server
        │
        ├── 新人指引 Agent → 通过 MCP 查询知识库
        ├── 出题 Agent → 通过 MCP 获取知识点和题目
        ├── 合规审查 Agent → 通过 MCP 检查文档规范性
        └── 第三方系统 → 通过 MCP 接入知识库能力
```

**MCP 的核心价值**：知识库不再是孤岛，任何 AI Agent 都可以通过标准化协议查询知识库，实现"知识即服务"。

## 四、行业竞品分析

### 4.1 主流开源方案对比

| 平台 | 核心定位 | 知识图谱 | GraphRAG | MCP 支持 | 适用场景 |
|------|---------|---------|----------|---------|---------|
| **Dify** | LLMOps 平台 | 不支持 | 不支持 | 支持 | 快速搭建 AI 应用，工作流编排 |
| **FastGPT** | AI 知识库 | 不支持 | 不支持 | 不支持 | 企业知识库问答，深度定制 |
| **RAGFlow** | 深度文档理解 RAG | 部分支持 | 实验性 | 不支持 | 高精度文档检索 |
| **Coze（字节）** | 智能体平台 | 不支持 | 不支持 | 不支持 | 快速搭建对话机器人 |
| **本方案** | 知识管理平台 | **Graphology** | **自建 TS 版** | **支持** | **政企知识管理 + 关联发现** |

### 4.2 差异化优势

现有方案（Dify / FastGPT / RAGFlow）都聚焦在"向量检索 + AI 问答"，**没有一家做知识图谱关联发现**。我们的差异化在于：

1. **GraphRAG 关联发现** — 查一个实体，自动发现所有关联实体和文档
2. **多格式统一管道** — Excel/PDF/Word 自动解析，不要求用户改格式
3. **MCP 开放接口** — 知识库作为基础设施，供其他 Agent 调用
4. **私有化部署** — 架构预留 Ollama 本地模型切换能力，数据不出内网

## 八、可落地性审视（个人开发者视角）

> 以下内容基于对大厂方案的调研和对开源项目（LightRAG、MCP SDK）的深入验证，
> 逐项审视原方案中哪些能落地、哪些需要调整、哪些应该砍掉。
> 最终方案：**纯 TypeScript 自建 GraphRAG**，单进程部署，不依赖 Python sidecar。

### 8.1 为什么不用 LightRAG / Python sidecar

LightRAG 是目前最轻量的 GraphRAG 开源实现，但它是 Python 库。集成到 Next.js 项目需要跑两个进程（Next.js + LightRAG Server），导致：

| 问题 | 影响 |
|------|------|
| **部署复杂度翻倍** | 不能用 Vercel 一键部署，需要 VPS + Docker Compose |
| **开发体验差** | 每次开发要同时启动两个服务，调试跨进程错误 |
| **面试官部署成本高** | clone 仓库后还需要 pip install + 配置 Python 环境 |
| **架构变重** | 一个演示项目搞成微服务，喧宾夺主 |

**结论**：Python sidecar 方案对部署的影响不可接受。采用路线 A — 纯 TypeScript 自建轻量 GraphRAG。

### 8.2 纯 TypeScript 自建 GraphRAG 方案

#### 核心思路

GraphRAG 的本质只有三步，不依赖任何 Python 库：

```
1. 实体提取：调 LLM API，从文本中提取实体和关系
   输入：一段文本
   输出：[{ entity: "隐患点A", type: "Location", relations: [{ target: "滑坡", type: "HAS_DISASTER_TYPE" }] }]

2. 图谱存储：实体和关系存入 JSON 邻接表（类似现有 FileStore）
   数据结构：{ nodes: Map<id, Entity>, edges: Map<id, Relation> }
   持久化：data/graph.json

3. 图谱查询：从查询中识别实体 → 遍历邻接表找关联实体 → 返回关联上下文
   查询："隐患点A的预警阈值" → 识别"隐患点A" → 找到关联的规范、人员、历史数据
```

#### 存储设计（已验证：Graphology 库）

经在线验证，使用 [Graphology](https://graphology.github.io/)（JS 图数据库库，npm 0.26.0）作为图存储引擎，
内置序列化/反序列化、BFS/DFS 遍历、度中心性等算法。演示级数据量（500 节点 / 2000 边）JSON 文件约 150KB，加载 < 5ms。

```typescript
// 依赖：graphology@0.26.0 + graphology-traversal@0.3.1
import Graph from 'graphology'
import { bfsFromNode } from 'graphology-traversal'

type NodeAttrs = { name: string; type: string; properties: Record<string, string>; sourceChunks: string[] }
type EdgeAttrs = { relation: string; evidence: string }

// 图谱实例
const graph = new Graph<NodeAttrs, EdgeAttrs>()

// 添加实体节点
graph.addNode('隐患点A', { name: '隐患点A', type: 'Location', properties: {}, sourceChunks: ['chunk-1'] })

// 添加关系边
graph.addEdge('隐患点A', '张三', { relation: 'RESPONSIBLE_FOR', evidence: '隐患点A的监测负责人为张三' })

// BFS 遍历查找关联实体（深度 2）
const related: string[] = []
bfsFromNode(graph, '隐患点A', (node, attr, depth) => {
  related.push(node)
  if (depth >= 2) return true // 停止从该节点继续扩展
})

// 持久化：graph.export() → JSON.stringify → data/graph.json
// 加载：JSON.parse → Graph.from(data)
```

#### 实体提取（已验证：Vercel AI SDK generateObject + Zod）

经在线验证，LLM 实体关系提取的 F1 值可达 **0.89-0.95**（[Lettria benchmark](https://www.lettria.com/benchmarks/benchmarking-text-to-graph)）。
使用 Vercel AI SDK 的 `generateObject` + Zod schema 实现类型安全的结构化提取，无需手写 Prompt 解析。

```typescript
// 依赖：ai@6.x + zhipu-ai-provider（智谱官方 AI SDK 适配器）
// 备选：@ai-sdk/openai + 自定义 baseURL 也可以
import { generateObject } from 'ai'
import { createZhipu } from 'zhipu-ai-provider'
import { z } from 'zod'

const zhipu = createZhipu({ apiKey: process.env.ZHIPU_API_KEY })

// GLM-5.1 关闭 thinking 模式（thinking 模式下 schema-guided 结构化输出不可用）
const extractionModel = zhipu('glm-5.1').withConfig({
  thinking: { type: 'disabled' }
})

// Zod schema — .describe() 的文字会作为提取提示发给 LLM
const extractionSchema = z.object({
  entities: z.array(z.object({
    name: z.string().describe('实体名称，如"隐患点A"、"张三"、"滑坡灾害防治规范"'),
    type: z.enum(['Person', 'Location', 'Organization', 'Document', 'DisasterType', 'Regulation', 'Equipment', 'Other']),
    description: z.string().describe('实体的简要说明'),
  })),
  relations: z.array(z.object({
    source: z.string().describe('源实体名称'),
    target: z.string().describe('目标实体名称'),
    relation: z.string().describe('关系类型，如 RESPONSIBLE_FOR / LOCATED_IN / HAS_DISASTER_TYPE / REFERENCES'),
    evidence: z.string().describe('支撑这条关系的原文片段'),
  })),
})

const { object } = await generateObject({
  model: extractionModel,
  schema: extractionSchema,
  prompt: `从以下文本中提取所有实体和关系：\n\n${chunkText}`,
})
```

#### 图谱可视化（已验证：react-force-graph-2d）

经在线验证，[react-force-graph-2d](https://github.com/vasturiano/react-force-graph-2d)（npm 1.29.1）是最快的 React 图谱可视化路径。
基于 HTML5 Canvas 2D 渲染，单组件即可渲染，数据格式与 Graphology 导出格式天然兼容。

```tsx
// 依赖：react-force-graph-2d@1.29.1
// Next.js 必须动态导入（因为依赖浏览器 Canvas API）
import dynamic from 'next/dynamic'
const ForceGraph2D = dynamic(() => import('react-force-graph-2d'), { ssr: false })

// Graphology 导出数据直接转换即可使用
const graphData = {
  nodes: Array.from(graph.nodeEntries()).map(([key, attrs]) => ({
    id: key, name: attrs.name, group: attrs.type,
  })),
  links: Array.from(graph.edgeEntries()).map(({ source, target, attributes }) => ({
    source, target, label: attributes.relation,
  })),
}

<ForceGraph2D
  graphData={graphData}
  nodeLabel="name"
  nodeAutoColorBy="group"
  linkDirectionalArrowLength={3}
/>
```

#### 图谱查询流程

```
用户查询: "隐患点A的预警阈值是多少？"
    │
    ├── Step 1: 从查询中识别关键实体
    │   方式 A：LLM 提取（准确但慢）
    │   方式 B：关键词匹配节点名称（快但粗）
    │   演示阶段用方式 B，生产用 A
    │   → 识别到实体: "隐患点A"
    │
    ├── Step 2: 图遍历（BFS，深度 2）
    │   隐患点A → [滑坡类型, XX市区域规范, 张三(负责人), 2025年监测数据]
    │   → 收集所有关联实体的 sourceChunks
    │
    └── Step 3: 与向量检索结果融合
        向量检索结果 + 图谱关联上下文 → 一起注入 LLM
```

#### 与现有代码的整合

```
现有架构（不动）：
  upload route → chunker → embedding → FileStore.addVectors()
  chat route   → embedding → FileStore.similaritySearch() → LLM

新增模块（插入到现有流程中）：
  lib/graph-store.ts     → GraphStore 类（类似 FileStore，JSON 持久化）
  lib/entity-extractor.ts → 调 LLM API 提取实体关系
  upload route 新增步骤  → 切片后调 entityExtractor → graphStore.addNode/Edge()
  chat route 新增步骤    → 图谱查询 → 关联实体加入上下文
```

#### 与 LightRAG 的能力对比

| 能力 | LightRAG（Python） | 自建方案（TypeScript） | 差距评估 |
|------|-------------------|---------------------|---------|
| 实体关系提取 | 内置，多层优化 | 一个 Prompt + JSON 解析 | 演示够用，准确率略低 |
| 图谱存储 | NetworkX / Neo4j | JSON 邻接表 | 演示级数据量无差异 |
| 检索模式 | local/global/hybrid/naive | 向量 + 图遍历 BFS | 覆盖最常用的 local 模式 |
| 增量更新 | 内置 | 上传时增量添加节点/边 | 同等效果 |
| 部署依赖 | Python 运行时 | **无额外依赖** | **核心优势** |

## 九、方案缺陷与风险分析

任何技术方案都有局限性，诚实面对这些问题比假装它们不存在更重要。以下是本方案的主要缺陷和应对策略。

### 9.1 技术层面的缺陷

| 缺陷 | 严重程度 | 说明 | 应对策略 |
|------|---------|------|---------|
| **LLM 幻觉无法根除** | 高 | 即使有 RAG 上下文，LLM 仍可能生成错误内容 | 来源追溯（已实现）+ 置信度标注 + 用户反馈机制 |
| **实体提取质量依赖 LLM** | 高 | LLM 提取的实体关系可能有遗漏或错误，污染图谱 | 人工审核机制 + 低置信度实体标记 + 定期清洗 |
| **知识图谱构建成本高** | 中 | 大量文档的实体提取需要多次 LLM 调用，耗时且费钱 | 增量构建（上传时逐步提取）+ 结构化数据直接导入（Excel）减少 LLM 调用 |
| **分块策略粗粒度** | 中 | 当前用字符级分块，可能切断语义完整的段落 | 升级为语义分块（基于句子边界 + LLM 判断段落完整性） |
| **FileStore / GraphStore 不适合生产** | 中 | JSON 文件存储无法支撑 10 万级并发查询 | 生产环境向量库切换 ChromaDB / Milvus，图谱切换 Neo4j |
| **多用户并发性能** | 中 | 无缓存层，每次查询都走全流程 | 添加 Redis 缓存热门查询 + 异步预加载 |

### 9.2 工程层面的缺陷

| 缺陷 | 严重程度 | 说明 | 应对策略 |
|------|---------|------|---------|
| **无增量更新机制** | 中 | 文档修改后需全量重新入库 | 实现文档版本管理 + 增量 diff 更新 |
| **权限控制粒度粗** | 高 | 当前无文档级权限，所有用户看到所有内容 | 实现文档级 ACL，搜索结果按权限过滤 |
| **无审计日志** | 中 | 缺少用户操作审计（谁查了什么、谁改了什么） | 添加操作日志 + 审计报告 |
| **监控告警缺失** | 低 | 无系统健康监控 | 添加 Prometheus + Grafana 监控 |

### 9.3 产品层面的缺陷

| 缺陷 | 严重程度 | 说明 | 应对策略 |
|------|---------|------|---------|
| **冷启动问题** | 高 | 系统上线初期知识库为空，无法展示价值 | 预置行业模板 + 快速导入工具 + 种子数据 |
| **用户习惯迁移** | 中 | 用户习惯了搜索文件夹，AI 对话模式需要适应 | 提供传统搜索 + AI 对话双入口 |
| **评估体系缺失** | 中 | 无法量化知识库的检索质量和回答准确性 | 实现 RAGAS 评估框架（答案相关性、上下文相关性、忠实度） |

### 9.4 与大厂方案的差距

| 维度 | Microsoft/Google | 本方案 | 差距说明 |
|------|-----------------|--------|---------|
| **数据源种类** | 20+ 种企业数据源 | 4 种（PDF/Word/Excel/TXT） | 邮件、代码、数据库可扩展（架构已预留） |
| **安全体系** | 企业级 DLP + 权限继承 | 基础 RBAC | 政府/金融场景必须补齐 |
| **评估体系** | A/B 测试 + 质量监控 | 无 | 必须添加 RAG 评估 |
| **规模** | 十亿级文档 | 万级 | 技术选型一致，主要是工程优化 |
